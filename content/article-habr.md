# One UI: как я собрал 52 React-компонента на Base UI, которые весят как один Button

> Черновик статьи для Habr. Ссылки на репозиторий, npm и демо помечены как `TODO` —
> подставьте перед публикацией. Цифры бандла реальные и воспроизводимы (см. конец статьи).

Большинство популярных UI-китов ставят вас перед выбором: либо красиво и «из коробки»,
но тяжело (MUI, Ant Design тащат CSS-in-JS и десятки килобайт на один компонент), либо
легко, но придётся самому рисовать каждую кнопку (headless-примитивы вроде Radix или
Base UI). Мне хотелось третьего варианта: доступные, типизированные, **готовые**
компоненты — но без рантайма стилей и без лишнего веса.

Так появился **One UI** — набор из 52 компонентов поверх [Base UI](https://base-ui.com),
со стилями на Tailwind (ноль рантайм-CSS), темой в пространстве OKLCH и фирменной
«желейной» микроанимацией. В этой статье — как он устроен, почему именно Base UI, и
честное сравнение размера бандла с MUI, Ant Design, Chakra, Mantine и Radix.

## TL;DR

- 52 компонента: от Button и Switch до Combobox, Drawer, Menu и Toast.
- Поверх Base UI — доступность (клавиатура, фокус, ARIA) достаётся бесплатно.
- **Ноль рантайм-CSS**: стили — это обычные слои Tailwind, скомпилированные на этапе сборки.
- Собственный код всех 52 обёрток — **10.8 кБ gzip**. Платите только за те примитивы
  Base UI, что реально импортируете.
- Тема — слой CSS-переменных в OKLCH; перекрасить всю систему можно парой токенов.

## Почему не «ещё один MUI»

Проблема тяжёлых китов не в том, что они плохие, а в их цене на входе. Когда вы пишете
`import Button from "@mui/material/Button"`, вы тянете не только кнопку, но и движок
стилей (Emotion), систему тем и кучу служебного кода. Для дизайн-систем это окупается,
но для лендинга или небольшого приложения — перебор.

Headless-примитивы (Radix, Base UI) решают проблему веса и доступности, но возвращают
вам пустой холст: каждый компонент нужно стилизовать с нуля. Это отличная основа для
своей дизайн-системы — но не «бери и пользуйся».

One UI — это тонкий, стилизованный слой поверх Base UI. Вы получаете готовый внешний вид
и поведение, но без рантайм-стоимости стилей.

## Архитектура: тонкие обёртки и data-атрибуты

Каждый компонент One UI — это очень тонкая обёртка над примитивом Base UI. Вся «магия»
в том, что варианты прокидываются как `data-*` атрибуты, а внешний вид задаётся в CSS
через Tailwind `@apply`. Вот как выглядит кнопка целиком:

```tsx
import { Button as BaseButton } from "@base-ui/react/button"
import { toClassNames, toDataAttrs } from "../../utils"

export const Button = (props: ButtonProps) => {
  const { variant = "primary", size = "md", iconOnly = false, className, ...rest } = props
  return (
    <BaseButton
      {...rest}
      {...toDataAttrs({ iconOnly, variant, size })}
      className={toClassNames("button", className)}
    />
  )
}
```

А стили — обычный CSS, который компилируется в статику на сборке:

```css
.button {
  @apply inline-flex items-center justify-center rounded-md font-medium transition;

  &[data-variant="primary"] {
    @apply bg-info text-info-foreground;
    &:hover { @apply bg-info-hover; }
  }
}
```

Никакого `styled()`, никакого пересчёта стилей в рантайме. Компонент остаётся
типизированным: `ButtonProps` расширяет пропсы Base UI, так что автокомплит не врёт.

Составные компоненты повторяют анатомию Base UI и экспортируются как namespace:

```tsx
import { Switch, Tabs } from "one-ui"

<Switch.Root defaultChecked>
  <Switch.Thumb />
</Switch.Root>

<Tabs.Root defaultValue="overview">
  <Tabs.List>
    <Tabs.Tab value="overview">Overview</Tabs.Tab>
    <Tabs.Tab value="activity">Activity</Tabs.Tab>
    <Tabs.Indicator />
  </Tabs.List>
  <Tabs.Panel value="overview">…</Tabs.Panel>
</Tabs.Root>
```

## Замеры бандла: честное сравнение

Любые сравнения веса легко подтасовать, поэтому я мерил всех одинаково: беру **один**
именованный импорт, бандлю изолированно через esbuild (`bundle + minify`, формат ESM),
выношу `react`/`react-dom` во external, затем сжимаю gzip. Одни и те же настройки для
каждой библиотеки. Все конкуренты реально установлены и измерены, а не взяты «по памяти».

Стоимость одного компонента (кБ, min + gzip, React исключён):

| Библиотека | Button | Switch | Dialog | Select |
| --- | --- | --- | --- | --- |
| **One UI** | **3.5** | **6.3** | **24.0** | **45.2** |
| Base UI (основа) | 3.0 | 6.0 | 22.4 | 43.8 |
| Radix UI | — | 3.4 | 11.6 | 26.8 |
| Mantine | 12.2 | 14.9 | 25.8 | 43.5 |
| Chakra UI | 13.2 | 21.9 | 31.1 | 44.8 |
| MUI | 36.8 | 36.7 | 37.5 | 53.3 |
| Ant Design | 55.9 | 52.4 | 85.4 | 91.7 |

Что из этого видно:

- **Обёртка One UI почти бесплатна.** Поверх Base UI она добавляет всего ~0.2–1.7 кБ.
- **Против CSS-in-JS китов разрыв кратный.** Button в One UI меньше MUI в ~10 раз и
  меньше Ant Design в ~16 раз. Плюс — ноль рантайм-CSS, тогда как Emotion-библиотеки
  платят ещё и за вычисление стилей в браузере.
- **Radix честно легче на оверлеях.** Это самый минималистичный headless-набор, и на
  Dialog/Select он выигрывает. Но он не даёт ни Button, ни готового внешнего вида — это
  другой уровень абстракции. Скрывать это нечестно, поэтому показываю как есть.

И ещё один важный нюанс. Если собрать весь публичный API One UI (все 52 компонента) с
`@base-ui/react` как внешней зависимостью, **собственный код библиотеки — всего 10.8 кБ
gzip**. Базовые примитивы ставятся отдельно и подтягиваются по мере импорта благодаря
tree-shaking. То есть вы не платите за то, чем не пользуетесь.

## Тема на OKLCH

Вся палитра — это слой CSS-переменных в перцептивном пространстве OKLCH. Семантические
токены (`info`, `success`, `warning`, `error`, `surface`, `neutral`) выводятся из
небольшого набора базовых, а тёмная тема — это класс `.dark` на родителе.

```css
:root {
  --color-info: oklch(0.62 0.19 254);   /* акцент бренда */
  --color-success: oklch(0.73 0.19 151);
}

.dark {
  --color-background: oklch(0.12 0.005 286);
  --color-foreground: oklch(0.99 0 0);
}
```

Перекрасить всю систему под бренд — это поменять несколько токенов, а не переопределять
стили каждого компонента.

## Фирменная деталь: «желейная» анимация

Мелочь, которая делает интерфейс живым. При взаимодействии элементы делают короткий
упругий «отскок» (squash & stretch) на чистом CSS:

```css
@keyframes jelly {
  0%   { scale: 1 1; }
  30%  { scale: 1.08 0.92; }
  40%  { scale: 0.95 1.05; }
  60%  { scale: 1.03 0.97; }
  100% { scale: 1 1; }
}
```

Отдельная история — тосты. Соседние уведомления используют зеркальный вариант
(`jelly-odd` со встречной фазой сжатия), чтобы стопка не пульсировала в унисон. Всё
уважает `prefers-reduced-motion`.

## Как попробовать

```bash
npm i one-ui @base-ui/react
```

```tsx
// один раз в корне приложения
import "one-ui/styles.css"
```

```tsx
import { Button, Switch } from "one-ui"

export function Example() {
  return (
    <div style={{ display: "flex", gap: 12, alignItems: "center" }}>
      <Button variant="primary">Get started</Button>
      <Switch.Root defaultChecked>
        <Switch.Thumb />
      </Switch.Root>
    </div>
  )
}
```

`react` и `react-dom` — peer-зависимости; «use client»-директивы сохраняются в сборке,
так что компоненты дружат с React Server Components.

## Воспроизводимость замеров

Чтобы цифрам можно было верить, замер оформлен скриптом. Ставите конкурентов как
dev-зависимости и запускаете:

```bash
npm run measure
```

Скрипт бандлит по одному импорту из каждой библиотеки одинаковыми настройками esbuild и
печатает таблицу gzip-размеров. Никакой магии — можете перепроверить сами.

## Итоги

One UI — попытка занять нишу между «тяжело, но готово» и «легко, но пусто»: вес
headless-подхода и ноль рантайм-CSS, но с готовым внешним видом, доступностью и приятной
анимацией из коробки. Если вам близок Base UI, но не хочется стилизовать 52 компонента
вручную — возможно, это сэкономит вам неделю.
